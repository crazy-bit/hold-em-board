## Context

对局详情页（`subpages/match/detail/detail.js`）的管理员点击"结束对局"时，会调用 `finishMatch()`。该方法目前只检查是否有未填写 `finalChips` 的成员，没有做数值合法性校验。

筹码守恒规则：所有玩家结算筹码总和 − 所有玩家额外加成总和 = 0。
- `scores[i].finalChips`：每个玩家填写的结算筹码
- `scores[i].bonus`：每个玩家本局额外加成（来自规则配置）

## Goals / Non-Goals

**Goals:**
- 在 `finishMatch()` 执行原有未填写检查之前，先做差额校验
- 差额不为 0 时弹出专用错误弹窗，展示差额数值，阻断结束流程
- 差额为 0（或全员未填写）时继续原有流程

**Non-Goals:**
- 不修改云函数 `finishMatch` 的服务端逻辑
- 不自动修正数值，只做提示
- 不对"部分人未填写"场景做差额计算（空值视为跳过）

## Decisions

### 决策 1：仅校验已填写的玩家

**选择**：只对 `finalChips !== null && finalChips !== undefined` 的成员求和计算差额。

**原因**：未填写者不影响数值守恒判断的语义——未填写是另一个维度的问题（已有提示），混在一起会产生误导性的差额值。若全员未填写则差额为 0，正常跳转原流程。

**备选**：将空值视为 0 参与计算——会导致误报，体验更差。

### 决策 2：前端校验，不走云函数

**选择**：在 `detail.js` 的 `finishMatch()` 方法中本地计算，不新增云函数调用。

**原因**：scores 数据已在页面 `this.data.scores` 中，无需额外网络请求；校验失败属于 UI 层拦截，不需要持久化。

### 决策 3：新增独立的差额错误弹窗

**选择**：新增 `showBalanceErrorModal` + `balanceDiff` 两个 data 字段，用一个独立弹窗展示差额，而不是复用未填写弹窗。

**原因**：两种错误性质不同（数据缺失 vs 数据错误），合并弹窗会让文案逻辑复杂；独立弹窗易于后续扩展。

## Risks / Trade-offs

- **精度问题**：筹码通常为整数，但若规则允许小数，浮点累加可能产生极小误差（如 0.0000001）。→ 使用 `Math.round` 或允许 `Math.abs(diff) < 0.01` 的容差判断。
- **数据时效**：页面 scores 数据在用户停留期间可能被其他人更新。→ 差额校验只作软提示，管理员仍可通过原"强制结束"路径继续（现有逻辑已有 `force: true`）；不做强制拦截。
