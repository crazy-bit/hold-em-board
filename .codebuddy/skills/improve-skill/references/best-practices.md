## 1. 简洁性

上下文窗口是共享资源。Skill 需要与系统提示、对话历史、其他 Skill 元数据以及实际请求竞争空间。

**默认假设：Claude 已经非常聪明。**

对每条信息都要质疑：
- "Claude 真的需要这个解释吗？"
- "这能被视为常识吗？"
- "这些内容值得消耗这些 token 吗？"

**好的示例**（约 50 个 token）：
```markdown
## 提取 PDF 文本

使用 pdfplumber 进行文本提取：

```python
import pdfplumber
with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
```

**差的示例**（约 150 个 token）— 在展示代码之前先解释什么是 PDF、有哪些库可用、以及为什么选择 pdfplumber。Claude 早已知道这些。

## 2. 适当的自由度

根据任务的脆弱性和可变性匹配具体程度。

| 自由度级别 | 适用场景 | 示例 |
|---|---|---|
| **高**（文字说明） | 存在多种有效方法；依赖上下文的决策 | 代码审查流程 |
| **中**（伪代码 / 参数化脚本） | 存在首选模式；允许一定变化 | 报告生成 |
| **低**（具体脚本，少量/无参数） | 脆弱操作；一致性至关重要；严格排序 | 数据库迁移 |

**类比：** 把 Claude 当作在路径上行走的机器人。
- 两侧是悬崖的窄桥 → 精确的护栏（低自由度）
- 没有危险的开阔地 → 大方向即可（高自由度）

## 3. YAML Frontmatter

SKILL.md frontmatter 中的两个必填字段：

**`name`：**
- 最多 64 个字符
- 仅限小写字母、数字和连字符
- 不含 XML 标签
- 不含保留词："anthropic"、"claude"

**`description`：**
- 非空
- 最多 1024 个字符
- 不含 XML 标签
- 描述 Skill 的功能**以及**何时使用

---

## 4. 命名规范

优先使用**动名词形式**（动词 + -ing）：
- `processing-pdfs`
- `analyzing-spreadsheets`
- `managing-databases`
- `testing-code`
- `writing-documentation`

可接受的替代形式：
- 名词短语：`pdf-processing`、`spreadsheet-analysis`
- 动作导向：`process-pdfs`、`analyze-spreadsheets`

避免：
- 模糊：`helper`、`utils`、`tools`
- 过于通用：`documents`、`data`、`files`
- 保留词：`anthropic-helper`、`claude-tools`

## 5. 有效的描述

description 字段驱动 Skill 的发现。

**始终使用第三人称：**
- ✅ "处理 Excel 文件并生成报告"
- ❌ "我可以帮你处理 Excel 文件"
- ❌ "你可以用这个来处理 Excel 文件"

**具体描述并包含关键词：**

好的示例：
```yaml
# PDF 处理
description: 从 PDF 文件中提取文本和表格，填写表单，合并文档。当处理 PDF 文件或用户提到 PDF、表单或文档提取时使用。

# Excel 分析
description: 分析 Excel 电子表格，创建数据透视表，生成图表。当分析 Excel 文件、电子表格、表格数据或 .xlsx 文件时使用。
```

## 6. 渐进式披露

三级加载系统：

1. **元数据**（name + description）— 始终在上下文中（约 100 词）
2. **SKILL.md 正文** — 触发 Skill 时加载（< 5000 词）
3. **捆绑资源** — 由 Claude 按需加载（无限制*）

*无限制是因为脚本可以在不加载到上下文的情况下执行。

**实践指南：**
- 保持 SKILL.md 正文在 500 行以内
- 接近上限时 → 将内容拆分到独立文件

**模式一：高层指南 + 参考资料**
```markdown
## 快速开始
[核心说明]

## 高级功能
**表单填写**：完整指南见 [FORMS.md](FORMS.md)
**API 参考**：所有方法见 [REFERENCE.md](REFERENCE.md)
```

**模式二：按领域组织**
按领域组织，避免加载无关上下文。

**模式三：条件性详情**
基础内容内联展示，高级内容链接到外部。

**避免深层嵌套** — 参考资料应从 SKILL.md 起只有一层深度。

**目录** — 超过 100 行的参考文件，在顶部加入目录。

## 7. 写作风格

- 使用**祈使/不定式形式**（动词开头的指令）
- 避免第二人称（"你应该"）
- 使用客观、指令性语言
  - ✅ "要完成 X，执行 Y"
  - ❌ "你应该做 X"
  - ❌ "如果你需要做 X"

## 8. 工作流与反馈循环

### 复杂任务工作流
将复杂操作分解为清晰的顺序步骤。对于特别复杂的工作流，提供检查清单：

```markdown
## 研究综合工作流

复制此清单并跟踪进度：

```
研究进度：
- [ ] 步骤 1：阅读所有源文档
- [ ] 步骤 2：识别关键主题
- [ ] 步骤 3：交叉验证观点
- [ ] 步骤 4：创建结构化摘要
- [ ] 步骤 5：核实引用
```
```

### 反馈循环
模式：**运行验证器 → 修复错误 → 重复**

此模式显著提升任何产出可验证产物的 Skill 的输出质量。

---

## 9. 内容指南

### 避免时效性信息
不要包含会过时的信息。如果必须包含历史背景，使用"历史说明"章节。

### 使用一致的术语
选定一个术语并在整个 Skill 中统一使用。不要在同义词之间交替使用。

---

## 10. 常见模式

### 模板模式
为输出格式提供模板。根据需求匹配严格程度：
- **严格**（API 响应、数据格式）：精确的模板结构
- **灵活**（通用指南）：合理的默认值，按需调整

### 示例模式
当输出质量依赖于示例时，提供输入/输出对。

### 条件工作流模式
通过清晰的分支逻辑引导 Claude 完成决策点。

## 11. Skill 中的可执行代码

### 解决问题，而非推卸责任
显式处理错误，而非推给 Claude：

```python
def process_file(path):
    """处理文件，如果不存在则创建。"""
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        print(f"文件 {path} 未找到，创建默认文件")
        with open(path, "w") as f:
            f.write("")
        return ""
```

### 工具脚本值得投入
即使 Claude 能编写脚本，预制脚本也能提供：
- 比生成代码更高的可靠性
- 节省 token
- 节省时间
- 跨使用场景的一致性

### 计划-验证-执行模式
以结构化格式创建计划 → 用脚本验证 → 执行。提前捕获错误。

### 包依赖
- **claude.ai**：可从 npm/PyPI 安装，也可从 GitHub 拉取
- **Claude API**：无网络访问，无运行时包安装

### MCP 工具引用
始终使用完全限定的工具名称：`ServerName:tool_name`

## 12. 应避免的反模式

| 反模式 | 修复方式 |
|---|---|
| Windows 风格路径（`scripts\helper.py`） | 使用正斜杠（`scripts/helper.py`） |
| 一个任务提供太多选项 | 展示一种推荐方式 |
| 假设工具已安装 | 明确列出依赖并验证可用性 |
| 脚本中的"魔法常量" | 为每个魔法值记录原因 |
| 深层参考嵌套 | 保持参考资料从 SKILL.md 起只有一层深度 |
| 解释 Claude 已知的内容 | 删除；假设 Claude 足够聪明 |
| 术语不一致 | 选定一个术语，统一使用 |

## 13. 评估与迭代

### 评估驱动开发
1. 识别差距：在没有 Skill 的情况下让 Claude 执行代表性任务，记录具体失败
2. 创建评估：构建三个测试这些差距的场景
3. 建立基线：衡量 Claude 在没有 Skill 时的表现
4. 编写最小化指令：仅创建足以弥补差距并通过评估的内容
5. 迭代：运行评估，与基线对比，持续优化

### 与 Claude 协作迭代
1. 在没有 Skill 的情况下完成任务
2. 识别可复用的模式
3. 让 Claude A 创建 Skill
4. 检查简洁性
5. 改进信息架构
6. 在类似任务上测试
7. 根据观察迭代

### 观察导航模式
观察 Claude 在实践中实际如何使用 Skill。记录它阅读了哪些章节、跳过了哪些，以及在哪里感到困惑。
