# 基于最佳实践改进 Skill

通过应用官方 Skill 编写最佳实践，系统性地改进现有 Skill。

## 前置条件

开始任何工作前，先加载以下参考资料：
- `references/best-practices.md` — 完整的编写原则目录
- `references/skill-anatomy.md` — Skill 结构、资源类型、渐进式披露
- `references/fix-priority.md` — 质量原则与优先修复分类

假设所有 Skill 内容——无论是新起草的还是已有的——均**未**按照最佳实践设计。

---

## 改进模式

1. **阅读 Skill** — 加载目标 SKILL.md 及所有捆绑资源
2. **分析** — `python3 scripts/analyze_skill.py <path>`
3. **人工审查** — 对照 `references/best-practices.md` 和 `references/fix-priority.md` 审查所有内容。自动分析捕捉结构问题；人工审查捕捉语气、简洁性、自由度校准和披露问题
4. **汇总** — 将自动化 + 人工发现合并为单一优先级列表
5. **修复** — 按优先级应用改进（严重 → 推荐 → 可选）
6. **重新分析** — `python3 scripts/analyze_skill.py <path>`
7. **验证** — `python3 scripts/quick_validate.py <path>`
8. **总结** — 呈现修改前后对比
9. **脚本化评估** — 识别 Skill 中可被脚本替代的步骤，给出替换建议

   对改进后的 Skill 逐步扫描，找出以下类型的候选步骤：
   - 重复性操作（每次执行路径固定、无需判断）
   - 结构化检查（格式校验、字段存在性、规则匹配）
   - 文件读写操作（路径固定、内容模板化）
   - 外部命令调用（shell 命令、工具链调用）

   对每个候选步骤，从三个维度评估：

   | 维度 | 评估标准 |
   |------|---------|
   | **稳定性** | 脚本逻辑是否确定？输入输出是否可预期？边界情况是否可枚举？ |
   | **收益** | 替换后节省多少 token / 时间？是否减少人工出错？执行频率高吗？ |
   | **维护性** | 脚本是否易读易改？依赖是否稳定？未来需求变化时改动成本高吗？ |

   输出评估报告，格式如下：

   ```
   ## 脚本化评估报告

   ### ✅ 建议脚本化（高收益）
   - 步骤 N：{步骤描述}
     - 稳定性：⭐⭐⭐ — {说明}
     - 收益：⭐⭐⭐ — {说明}
     - 维护性：⭐⭐⭐ — {说明}
     - 建议：{具体替换方案，如脚本名称、参数设计}

   ### ⚠️ 可选脚本化（中等收益）
   - 步骤 N：{步骤描述}
     - ...
     - 建议：{替换方案或保留理由}

   ### ❌ 不建议脚本化（低收益或高风险）
   - 步骤 N：{步骤描述}
     - 原因：{说明为何保留人工执行更合适}
   ```

   评估完成后，询问用户是否需要实现其中某个脚本。

## 内置脚本

| 脚本 | 用途 |
|---|---|
| `analyze_skill.py` | 自动化最佳实践分析 |
| `quick_validate.py` | 快速验证 frontmatter 和结构 |
| `package_skill.py` | 将 Skill 打包为可分发的 zip |

## 最终确认

在最终确定任何 Skill 之前，对照 `references/checklist.md` 进行核查。复制清单并在审查时逐项标记。
