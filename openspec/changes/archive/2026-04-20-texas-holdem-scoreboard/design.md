## Context

这是一个全新的微信小程序项目，用于德州扑克线下局的积分统计。目前没有现有系统，玩家依赖手动记录。项目需要支持多人协作填写分数、管理员管理赛程、灵活的积分规则配置。

约束条件：
- 微信小程序生态，需要微信登录
- 需要实时性（组员填写分数后其他人可见）
- 数据需要持久化存储

## Goals / Non-Goals

**Goals:**
- 提供便捷的记分组创建和加入方式（链接/二维码）
- 管理员可完整管理赛程生命周期（创建→进行→结束/销毁）
- 组员可自助填写分数，减少管理员负担
- 支持灵活的积分规则配置（初始筹码、额外加成）
- 提供清晰的赛程和总积分查看界面

**Non-Goals:**
- 不做实时对战/在线德州扑克游戏
- 不做复杂的数据分析和图表
- 不做跨平台（仅微信小程序）
- 不做支付/充值功能
- 第一版不做消息推送通知

## Decisions

### 1. 技术栈选择：微信小程序 + 云开发

**选择**: 使用微信小程序原生框架 + 微信云开发（CloudBase）

**理由**:
- 云开发免去服务器运维，自带数据库、云函数、存储
- 微信登录天然集成，无需额外认证系统
- 小程序分享链接和二维码能力原生支持
- 适合小型工具类应用，成本低

**备选方案**:
- Taro + 独立后端：跨平台但增加复杂度，当前不需要
- uni-app：生态不如原生微信小程序

### 2. 数据库设计：云数据库（MongoDB-like）

**选择**: 使用微信云开发数据库，文档型存储

**集合设计**:
- `groups` - 记分组（名称、管理员、规则配置、邀请码）
- `group_members` - 组成员关系（groupId、userId、加入时间）
- `matches` - 赛程（groupId、状态、创建时间）
- `scores` - 分数记录（matchId、userId、初始筹码、额外加成、结算筹码）

**理由**:
- 云数据库免运维，自动扩容
- 文档型结构灵活，适合规则配置等半结构化数据
- 支持实时数据推送（watch）

### 3. 积分计算逻辑

**选择**: 积分 = 结算筹码 - 初始筹码 + (额外加成是否计入 ? 额外加成 : 0)

**规则配置存储在 group 文档中**:
- `chipRules`: 数组，按名次定义初始筹码和额外加成
- `bonusCountsToTotal`: 布尔值，额外加成是否记入总积分

### 4. 赛程状态机

**选择**: 赛程状态流转：`active` → `finished` | `cancelled`

- `active`: 创建后，组员可填写分数
- `finished`: 管理员结束，分数录入总积分
- `cancelled`: 管理员销毁，分数作废不计入总积分

### 5. 邀请机制

**选择**: 生成唯一邀请码，支持小程序码和分享链接两种方式

**理由**: 微信小程序原生支持生成带参数的小程序码，分享链接可通过小程序分享卡片实现

## Risks / Trade-offs

- **[并发写入]** 多人同时填写分数可能冲突 → 每人只能写自己的分数记录，通过 userId 隔离
- **[数据一致性]** 结束赛程时需要确保所有人已填写 → 管理员结束前展示未填写人员列表，由管理员决定是否强制结束
- **[规则变更影响]** 修改规则后历史赛程是否受影响 → 规则变更仅影响后续赛程，历史赛程保留当时的规则快照
- **[云开发限制]** 免费额度有限 → 初期用户量小，免费额度足够；后续可升级付费套餐
